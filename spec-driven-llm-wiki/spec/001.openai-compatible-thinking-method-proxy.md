# Spec 001: OpenAI-Compatible Multi-Pass Thinking Method Proxy

## Task/Prompt

Buat spec untuk parent project `thinking-method` agar tersedia local OpenAI-compatible server di `localhost:30000` yang bisa dipakai OpenCode. Server harus mendukung pemilihan provider, minimal provider `minimax`, logging prompt masuk dan response keluar, serta pipeline multi-pass yang menjalankan beberapa prompt/provider call secara berurutan sebelum jawaban final dikembalikan ke user.

Requirement penting: ini bukan sekadar menambahkan satu system prompt berisi semua metode berpikir. Alur yang dimaksud adalah:

```text
prompt user
  -> thinking step 1 -> hasil 1
  -> thinking step 2 -> hasil 2
  -> thinking step 3 -> hasil 3
  -> ...
  -> thinking step N -> hasil N
  -> final aggregation/answer
  -> baru response dikembalikan ke user
```

Setiap step harus memakai prompt asli plus hasil step sebelumnya sebagai konteks. Semua prompt step, hasil step, metrics, dan final answer harus dilog agar bisa dilihat kembali dan dipakai sebagai training data.

Metode berpikir wajib untuk pipeline MVP berjumlah 10 provider calls:

1. Problem Framing
2. First Principles Thinking
3. Socratic Thinking
4. Root Cause Thinking
5. Systems Thinking
6. Hypothesis-Driven Thinking
7. Inversion Thinking
8. Trade-off Thinking
9. Second-Order Thinking
10. Probabilistic Thinking

`Decision Matrix`, `Synthesis`, dan `Answer` tidak dihitung sebagai thinking call default. Jawaban final dibuat oleh final aggregation setelah 10 call selesai.

## Tujuan

- Menyediakan endpoint lokal OpenAI-compatible sehingga OpenCode cukup diarahkan ke `http://localhost:30000/v1`.
- Memisahkan HTTP adapter, provider registry, provider implementation, multi-pass orchestration, dan logging supaya provider lain bisa ditambah tanpa mengubah route utama.
- Mendukung provider awal `minimax` dengan API key dan base URL dari environment variable.
- Menjalankan prompt melalui multi-pass thinking pipeline: step 1 menghasilkan output 1, step 2 membaca output 1, dan seterusnya sampai semua step terkonfigurasi selesai.
- Mengembalikan response ke user hanya setelah seluruh thinking step dan final aggregation selesai.
- Mencatat prompt asli, prompt tiap step, hasil tiap step, final response, metrics, dan error untuk debugging serta training data.
- Menjaga secret tidak pernah hardcoded dan tidak pernah muncul di log.

## Kenapa Tujuan Ini

OpenCode biasanya mudah diintegrasikan dengan endpoint OpenAI-compatible. Dengan membuat proxy lokal, project bisa melakukan orchestration multi-pass sebelum response dikembalikan, memilih provider seperti Minimax, dan mengamati seluruh jejak prompt/response tanpa perlu mengubah OpenCode. Arsitektur proxy juga menjaga eksperimen provider dan pengumpulan training data tetap terpusat di satu layer.

## Logic Changes

### 1. Runtime Configuration

Tambahkan konfigurasi parent project melalui `.env` dan loader typed config:

```text
HOST=127.0.0.1
PORT=30000
DEFAULT_PROVIDER=minimax
DEFAULT_MODEL=<minimax-default-model>
MINIMAX_API_KEY=<secret>
MINIMAX_BASE_URL=<minimax-api-base-url>
LOG_LEVEL=info
LOG_FORMAT=text
LOG_DIR=logs
LOG_INCLUDE_CONTENT=true
THINKING_PIPELINE_ENABLED=true
THINKING_STEP_COUNT=10
THINKING_RETURN_MODE=final
TRAINING_TRACE_ENABLED=true
```

Aturan:

- Jika `MINIMAX_API_KEY` kosong dan provider terpilih `minimax`, request harus gagal dengan error konfigurasi yang jelas.
- Nilai secret harus disanitasi dari response error dan log.
- `LOG_INCLUDE_CONTENT` menentukan apakah full prompt/response dicatat atau hanya metadata dan preview.
- `THINKING_PIPELINE_ENABLED=false` mematikan multi-pass orchestration dan langsung meneruskan request ke provider.
- `THINKING_STEP_COUNT=10` berarti request user diproses melalui 10 provider calls berurutan sebelum final aggregation.
- `THINKING_RETURN_MODE=final` berarti user hanya menerima final answer; intermediate step tetap tersimpan di log/training trace.
- `TRAINING_TRACE_ENABLED=true` membuat trace tiap step ditulis ke log/file terstruktur agar bisa dipakai ulang sebagai data training.

### 2. OpenAI-Compatible HTTP Layer

Endpoint minimum:

- `GET /health` mengembalikan status server dan provider registry readiness.
- `GET /v1/models` mengembalikan daftar model OpenAI-compatible.
- `POST /v1/chat/completions` menerima request OpenAI Chat Completions.

Kontrak kompatibilitas minimum untuk `POST /v1/chat/completions`:

- Input mendukung `model`, `messages`, `temperature`, `top_p`, `max_tokens`, `stream`, dan field umum lain yang bisa diteruskan secara best-effort.
- Output non-streaming mengikuti bentuk:

```json
{
  "id": "chatcmpl-...",
  "object": "chat.completion",
  "created": 1710000000,
  "model": "minimax/<model>",
  "choices": [
    {
      "index": 0,
      "message": { "role": "assistant", "content": "..." },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 0,
    "completion_tokens": 0,
    "total_tokens": 0
  }
}
```

Error response mengikuti pola OpenAI-compatible:

```json
{
  "error": {
    "message": "Sanitized error message",
    "type": "invalid_request_error",
    "param": null,
    "code": "provider_config_missing"
  }
}
```

### 3. Provider Selection

Provider resolution dilakukan sebelum prompt transformation dikirim upstream.

Aturan resolusi:

- Jika `model` berbentuk `provider/model`, gunakan `provider` sebagai provider id dan sisanya sebagai upstream model.
- Jika `model` tidak memuat prefix provider, gunakan `DEFAULT_PROVIDER` dan `model` sebagai upstream model.
- Jika `model` kosong, gunakan `DEFAULT_PROVIDER` dan `DEFAULT_MODEL`.
- Jika provider tidak dikenal, return OpenAI-compatible error `provider_not_found`.

Contoh:

```text
model: "minimax/abab6.5s-chat" -> provider=minimax, upstream_model=abab6.5s-chat
model: "abab6.5s-chat" -> provider=DEFAULT_PROVIDER, upstream_model=abab6.5s-chat
model: "" -> provider=DEFAULT_PROVIDER, upstream_model=DEFAULT_MODEL
```

### 4. Minimax Provider

Provider `minimax` harus mengimplementasikan interface provider internal:

```text
Provider.chatCompletion(request) -> OpenAICompatibleChatCompletion | AsyncIterable<OpenAICompatibleChunk>
Provider.listModels() -> OpenAICompatibleModel[]
Provider.health() -> ProviderHealth
```

Mapping wajib:

- OpenAI-compatible `messages` ke format messages Minimax.
- Parameter umum seperti `temperature`, `top_p`, `max_tokens`, dan `stream` diteruskan jika didukung Minimax.
- Response Minimax dinormalisasi menjadi schema OpenAI-compatible.
- Token usage dipetakan jika provider mengembalikan usage; jika tidak tersedia, field boleh `0` atau omitted sesuai schema internal.

Batasan:

- Spec ini tidak mengunci exact Minimax endpoint dan model default karena perlu dikonfirmasi dari kredensial/API yang tersedia saat implementasi.
- Implementasi wajib menaruh base URL dan API key di `.env.example`, bukan hardcoded.

### 5. Multi-Pass Thinking Pipeline

Pipeline berjalan setelah provider/model dipilih. Berbeda dari pendekatan satu prompt besar, pipeline ini melakukan beberapa provider call secara berurutan. Setiap step punya prompt khusus, menghasilkan output khusus, dan output itu menjadi input untuk step berikutnya.

Aturan utama:

- Preserve semua `messages` asli dari client sebagai `original_messages`.
- Ekstrak user task dari request asli tanpa menghapus context system/developer yang relevan.
- Jalankan `THINKING_STEP_COUNT` step secara serial, bukan paralel, karena step berikutnya bergantung pada output step sebelumnya.
- Default `THINKING_STEP_COUNT=10` menjalankan 10 thinking pass sebelum final aggregation.
- Setiap step request ke provider memakai:
  - prompt asli user,
  - context system/developer yang aman,
  - nama step,
  - instruksi step,
  - daftar hasil step sebelumnya.
- Setiap step menghasilkan structured result dengan minimal field:
  - `step_index`
  - `step_name`
  - `step_prompt`
  - `step_output`
  - `provider`
  - `model`
  - `usage`
  - `timing`
  - `success`
- Jika satu step gagal, pipeline harus log error step tersebut dan return OpenAI-compatible error kecuali konfigurasi explicit mengizinkan partial result.
- Setelah semua step selesai, jalankan final aggregation call untuk menyusun jawaban akhir dari semua hasil step.
- User menerima final answer saja secara default; semua intermediate result tetap tersimpan di log/training trace.

Default 10-step plan:

```text
1. Problem Framing
2. First Principles Thinking
3. Socratic Thinking
4. Root Cause Thinking
5. Systems Thinking
6. Hypothesis-Driven Thinking
7. Inversion Thinking
8. Trade-off Thinking
9. Second-Order Thinking
10. Probabilistic Thinking
```

Deferred optional extended analysis:

```text
Decision Matrix Thinking
Synthesis
```

Untuk kebutuhan yang diklarifikasi user, MVP wajib menjalankan 10 provider calls berurutan dulu. Extended analysis boleh ditambahkan nanti, tetapi tidak boleh mengubah requirement utama bahwa default response menunggu 10 thinking outputs selesai lalu final aggregation membuat `Answer`.

Template prompt tiap step:

```text
You are executing one step in a multi-pass thinking pipeline.

Original user request:
{{original_user_prompt}}

Previous step results:
{{previous_step_results}}

Current step:
{{step_index}}. {{step_name}}

Task:
- Focus only on this current thinking method.
- Produce a concise but complete structured result for this step.
- Do not produce the final answer unless this is the final aggregation step.
- Make the output useful as training data.

Return format:
<STEP_RESULT step="{{step_index}}" name="{{step_name}}">
...
</STEP_RESULT>
```

Template final aggregation:

```text
You are the final aggregator.

Original user request:
{{original_user_prompt}}

All step results:
{{all_step_results}}

Task:
- Read all completed step results.
- Synthesize them into the strongest final answer.
- Do not invent facts not supported by the original request or step results.
- Return only the final user-facing response unless debug return mode is enabled.

Return format:
<ANSWER>
...
</ANSWER>
```

### 6. Logging Flow

Logging mengikuti `LOGGING_DESIGN.md` dan interface yang sudah ada.

Urutan log per request:

1. `PROMPT` incoming: request id, provider/model target awal, messages asli, request metadata.
2. `CONFIG` provider resolution: provider final, upstream model, pipeline enabled/disabled, step count, return mode.
3. `PROMPT` step input: prompt yang dikirim untuk setiap thinking step.
4. `RESPONSE` step output: hasil provider untuk setiap thinking step.
5. `METRICS` step metrics: latency dan token usage untuk setiap step.
6. `PROMPT` final aggregation input: semua hasil step yang akan disintesis.
7. `RESPONSE` final answer: jawaban akhir yang dikembalikan ke user.
8. `METRICS` request metrics: total latency, total provider calls, total token usage jika tersedia.
9. `ERROR`: error code, sanitized message, provider id, step index jika error terjadi di step tertentu, request id.

Training trace wajib memuat record terstruktur:

```json
{
  "request_id": "req_...",
  "original_messages": [],
  "provider": "minimax",
  "model": "abab6.5s-chat",
  "steps": [
    {
      "step_index": 1,
      "step_name": "Problem Framing",
      "prompt": "...",
      "output": "...",
      "usage": {},
      "timing": {},
      "success": true
    }
  ],
  "final_prompt": "...",
  "final_answer": "...",
  "total_usage": {},
  "created_at": "2026-05-10T00:00:00.000Z"
}
```

Sanitization wajib:

- Redact API key, authorization header, cookies, bearer token, dan environment secret.
- Optional truncate content berdasarkan `max_content_length`.
- Jangan log raw stack trace ke client; stack trace hanya boleh masuk log debug jika aman.
- Untuk kebutuhan training data, raw prompt/output boleh disimpan hanya jika `TRAINING_TRACE_ENABLED=true` dan secret redaction tetap aktif.

### 7. Streaming

OpenCode dapat membutuhkan streaming untuk UX terbaik, tetapi multi-pass pipeline membuat streaming lebih kompleks karena user sebaiknya tidak menerima jawaban sebelum semua step selesai.

Aturan:

- MVP wajib menerima `stream: false` dan menghasilkan full final answer setelah semua step selesai.
- Jika request memakai `stream: true`, MVP boleh menahan stream sampai final answer siap lalu mengirim final answer sebagai stream OpenAI-compatible.
- Intermediate step tidak boleh dikirim ke user secara default karena user meminta semua thinking selesai dulu baru kembali.
- Debug mode boleh mengirim progress event internal hanya jika endpoint khusus atau config eksplisit disediakan, bukan default OpenCode path.
- Full streaming support memakai Server-Sent Events dengan chunk `chat.completion.chunk` dan terminator `data: [DONE]`.

## Pseudocode

```text
main:
  config = loadEnv()
  logger = createLogger(config.logging)
  registry = new ProviderRegistry()
  registry.register(new MinimaxProvider(config.minimax, logger))
  orchestrator = new ThinkingOrchestrator(config.pipeline, registry, logger)
  traceWriter = new TrainingTraceWriter(config.training, logger)
  server = createHttpServer(config, registry, orchestrator, traceWriter, logger)
  server.listen(config.host, config.port)

GET /health:
  provider_statuses = registry.healthAll()
  return { status: "ok", providers: provider_statuses }

GET /v1/models:
  models = registry.listModels()
  return { object: "list", data: models }

POST /v1/chat/completions:
  request_id = generateRequestId()
  received_at = now()
  body = parseAndValidateOpenAIChatRequest(req.body)
  log PROMPT incoming body.messages

  resolution = resolveModel(body.model, config.defaults, registry)
  if resolution invalid:
    log ERROR provider/model error
    return openAIError(resolution.error)

  log CONFIG provider resolution

  try:
    if config.pipeline.enabled:
      pipeline_result = orchestrator.run({
        request_id,
        original_request: body,
        provider: resolution.provider,
        upstream_model: resolution.upstream_model,
        step_count: config.pipeline.step_count
      })

      for each step_result in pipeline_result.steps:
        log PROMPT step_result.step_prompt
        log RESPONSE step_result.step_output
        log METRICS step_result.usage and step_result.timing

      log PROMPT pipeline_result.final_prompt
      log RESPONSE pipeline_result.final_answer
      log METRICS pipeline_result.total_usage and total timing

      if config.training_trace.enabled:
        traceWriter.write({
          request_id,
          original_messages: body.messages,
          provider: resolution.provider.id,
          model: resolution.upstream_model,
          steps: pipeline_result.steps,
          final_prompt: pipeline_result.final_prompt,
          final_answer: pipeline_result.final_answer,
          total_usage: pipeline_result.total_usage
        })

      normalized = buildOpenAIChatCompletionFromFinalAnswer(pipeline_result.final_answer, resolution, pipeline_result.total_usage)
      if body.stream:
        return writeBufferedFinalAnswerAsSse(normalized, logger, request_id)
      return normalized

    provider_response = provider.chatCompletion({
      ...body,
      model: resolution.upstream_model,
      messages: body.messages
    })
    normalized = normalizeProviderResponse(provider_response, resolution)
    log RESPONSE direct provider content
    log METRICS direct provider usage and timing
    return normalized
  catch error:
    safe_error = sanitizeProviderError(error)
    log ERROR safe_error
    return openAIError(safe_error)
```
